# MedHead PoC -- Reporting Document

> **Audience:** the MedHead Consortium architecture committee.
> **Purpose:** the public deliverable that the architecture committee evaluates the PoC against. Every claim is backed by a public artefact (source path, test path, Gherkin scenario, CI workflow step, JMeter report, Postman collection, Dockerfile, `readme.md` section) reproducible from a fresh checkout.
> **PoC scope reminder:** real-time recommendation of the nearest hospital offering an available bed in the required specialty -- a RESTful API with a graphical front-end, sized as the first item on the consortium's roadmap. The PoC is item #1 on the roadmap; every subsequent capability depends on it shipping cleanly. The PoC ships at the merge of this document.

**Status vocabulary used in §2 and §3.** Every Status cell in the tables below carries one of exactly four values:

- **Compliant** -- fully applied, with verifiable evidence.
- **Compliant -- C4-IV non-relaxed** -- the same as Compliant, with the inline reminder that B1, B2, B3, B4 are never relaxed for a PoC. Carried on those four rows only.
- **C4-III relaxation** -- explicit relaxation declared up-front, tagged with the decision ID and the production target it relaxes from. Carried on exactly two cells across the whole document: STD-8 (OpenID Connect, relaxed to simplified JWT under V3) and B5 (Shift-Left Security, relaxed under V3 / C4-III).
- **Justified deviation** -- intentional non-application with rationale and a production-path entry.

## §5. Lessons learned

Seven lessons drawn from the development process. Each is tied to a phase of the PoC and grounded in a concrete public artefact.

1. **Inside-out hexagonal TDD plus an outer-loop Gherkin scenario produces a usable pyramid without rework.** The eight back-end vertical slices each shipped tests + impl in the same commit, every PR green on the required CI checks before merge, never red on `master`. The pyramid in §4.2 (91 >> 39 >> 6) is the direct artefact of that discipline -- not the result of a retroactive coverage exercise. Citation: `backend/src/test/resources/features/`, `backend/src/test/java/com/medhead/poc/`, `.github/workflows/backend-ci.yml`.

2. **Measure-then-add is cheaper than add-then-justify.** Phase A's baseline FAIL on every dimension (read aggregate 497.9 req/s, p95 487 - 971 ms, write 503 share 22.0%) drove the decision to add Caffeine; Phase B verified the win (read aggregate 863.2 req/s, `/specialties` p95 1 ms, write 503 share 9.2%). The reporting argument "measured X, added cache, remeasured Y" is structurally stronger than "added because the rubric recommends it" and turns caching from a prescribed item into evidence (decision D21). Citation: `backend/jmeter/phase_a_baseline.md`, `backend/jmeter/phase_b_rerun.md`, `backend/src/main/java/com/medhead/poc/infrastructure/config/CachingConfig.java`.

3. **Port-level caching keeps the domain pure while still satisfying the C3 caching requirement.** `@Cacheable` and `@CacheEvict` live on the driven-port adapter (`backend/src/main/java/com/medhead/poc/infrastructure/adapter/out/persistence/`), not on domain services -- so the domain layer remains framework-free, the eviction-on-write contract is testable in isolation (`backend/src/test/java/com/medhead/poc/infrastructure/adapter/out/persistence/BedReservationCacheEvictionIT.java`), and the swap-Caffeine-for-Redis path stays a one-bean change in `CachingConfig`. Citation: `backend/src/main/java/com/medhead/poc/infrastructure/config/CachingConfig.java`, `backend/src/test/java/com/medhead/poc/infrastructure/adapter/out/persistence/BedReservationCacheEvictionIT.java`.

4. **The singleton Testcontainers pattern is mandatory when Spring's context cache reuses contexts across IT classes.** `@Testcontainers` + `@Container` restarts the container per class on a new random port, while Spring's context cache reuses the same context across IT classes -- stranding HikariCP on a dead port. The trap surfaced only on the second cached MockMvc IT and is the kind of latent issue an SBB consumer has to be warned about up front. Citation: `backend/src/test/java/com/medhead/poc/support/AbstractIntegrationIT.java` (singleton `static POSTGRES.start()` pattern).

5. **Vitest over Jest because Vite-native.** `frontend/vite.config.ts` carries one config for build + test; the same TypeScript / asset pipeline drives both; the Jest + ESM + Vite interop ceremony (ts-jest or @swc/jest, Babel ESM shim, CSS / asset import mocks) disappears for zero loss of test-authoring API (decision D22). Citation: `frontend/vite.config.ts`, `frontend/package.json` (`vitest`, `jsdom`, `@vitest/coverage-v8`).

6. **Joint-evidence BDD scoring -- backend Cucumber + frontend RTL user-centric specs -- is enough for C3 / C4 BDD compliance.** A second cross-stack browser layer (Cypress / Playwright + Gherkin) would duplicate the same evidence at twice the maintenance cost. The 6 backend scenarios cover the §2.6 main scenario, the fallback (risk #3), 400 + 404 error envelope, and both reference reads; the frontend RTL outer specs cover the same user behaviours in the user's language. Citation: `backend/src/test/resources/features/`, `frontend/src/recommend/RecommendForm.test.tsx`.

7. **The SBB-handoff discipline -- name every portable artefact with copy-customise points -- is what makes the PoC reusable beyond this committee's grading.** Each of `.github/workflows/backend-ci.yml`, `.github/workflows/frontend-ci.yml`, `backend/Dockerfile`, `backend/jmeter/`, `postman/`, `frontend/vite.config.ts` is a self-contained piece a sister team can copy and customise; the `readme.md` "CI/CD Pipeline" section names the four customisation surfaces (the `.feature` files, the Postman requests, the JMX thread groups, the seed `data.sql`).

### §6.1 Phase 1 -- Code (sessions S2 + S5 + S6)

**What the repository evidences.** Inside-out hexagonal TDD on the back-end (eight vertical slices, every PR shipping tests + impl in the same commit); outer-loop BDD authoring one Gherkin scenario per vertical slice (`backend/src/test/resources/features/`); a parallel front-end stream with user-centric RTL specs as the front-end's BDD evidence (`frontend/src/recommend/RecommendForm.test.tsx`); never-commit-red rule enforced by branch protection on `master` (six required job-level checks per `readme.md` "CI/CD Pipeline").

**Team support an architect would provide.**
- Pair-programming during the inside-out hexagonal phase to seed the discipline -- the domain port, then the unit test, then the adapter, in that order. The pattern is teachable in two PRs.
- A daily 15-minute architecture stand-up the first sprint of every new bounded context, then weekly once the rhythm is set.
- A locked decision-rhythm: ambiguities surface as kickoff resolutions before the first PR of the session, not mid-implementation. The decision IDs (V1-V6, D1-D22) are the public trail; in production they become Architecture Decision Records on the same shape.
- Conventional commits + branch naming from day one -- the per-feature traceability in `readme.md` "Branch workflow" is what makes a green PR also a release-noteable change.

### §6.2 Phase 2 -- Testing (sessions S5 + S6 + S7)

**What the repository evidences.** Three-level pyramid (91 unit >> 39 integration >> 6 E2E); BDD as ubiquitous-language acceptance, not as a brittle browser-driven E2E; data-driven non-functional decisions through Phase A baseline + Phase B rerun (`backend/jmeter/phase_a_baseline.md`, `backend/jmeter/phase_b_rerun.md`); coverage observed (97.2% / 71.9% backend lines / branches; 97.0% / 87.6% frontend) without build-time gating; Newman-runnable Postman collection (`postman/`).

**Team support an architect would provide.**
- A mob-review on the first SLA-significant PR (here it would have been the recommendation endpoint or the caching slice); the `Testcontainers @Container vs Spring context-cache` trap (lesson #4) is exactly the kind of failure mode the architect surfaces in mob-review before a junior engineer is alone with it.
- A weekly test-strategy review while the surface is growing -- pyramid shape, test-runtime budget, gap-versus-disposition trade-offs.
- A locked "never commit red" + the `failsafe:verify` boundary as the team's testing rhythm: unit tests fail fast, integration tests run on every push, E2E tests run on `verify`. Each layer answers a different question; mixing them is a smell.
- BDD framed as ubiquitous-language acceptance from day one -- otherwise teams write Gherkin as English UI tests and the C3 / C4 compliance argument decays.

### §6.3 Phase 3 -- CI/CD (sessions S3 + S8)

**What the repository evidences.** S3 minimum guardrail before any feature code -- two workflows on `master` with branch protection on day one; S8 hardening once the test surface stabilised -- multi-stage pipeline (`test` -> `build` -> `docker` for back-end; `lint` ‖ `test` -> `build` for front-end); concurrency-keyed cancellation; Docker smoke as the deployable-image gate; `readme.md` "CI/CD Pipeline" as the runbook.

**Team support an architect would provide.**
- A single named owner accountable for the runbook; in production this is the platform-engineering lead. The runbook is a living document, not an artefact frozen at S8 close.
- Pipeline runtime budget (back-end under 10 minutes, front-end under 2 minutes) treated as a first-class SLI alongside the API response-time SLA. Drift on the runtime budget is a leading indicator of test-pyramid drift.
- A quarterly pipeline review against drift -- new flaky tests, new heavy steps, new artefact uploads. The required-checks list in `readme.md` "Branch workflow" is the contract; everything outside it is opt-in.
- Branch protection as a non-negotiable: red cannot reach `master`, force-push is blocked, branch deletion is blocked. The team learns the rhythm because the runtime enforces it.

### §6.4 Phase 4 -- Scaling (post-PoC)

**What this PoC pre-clears.** Each of the production extensions named in §2 + §3 + §4 + §5 is a bounded follow-on item with a public seam:

- Resilience4j circuit breaker + bulkhead on the OSRM client (STD-7 + B1).
- Service mesh (Istio / Linkerd / Consul Connect) once the architecture grows to two or more services (STD-3).
- OpenID Connect swap (STD-8 + B5): `JwtDecoder` bean reconfiguration plus user-store replacement -- nothing else moves.
- Kafka or RabbitMQ event publisher (STD-1 + B6) -- replaces `SpringBedReservationEventPublisher`, port unchanged.
- `RedisCacheManager` replacing `CaffeineCacheManager` (D13) -- cache bean names + annotations stay.
- Pact-based consumer-driven contracts (B4) when downstream services come online.
- SAST + DAST + dependency scanning + threshold-gated jobs in CI (B5).
- Source the hospital register from the Organisation Data Service and the specialty taxonomy from the NHS Data Dictionary; reseed via Flyway or Liquibase instead of `backend/src/main/resources/data.sql` (D6 + D20).
- Cloud deployment per NHS Public-Cloud-First (NHS-4): Helm or Terraform module, managed Postgres + Redis, horizontal scaling behind a load balancer, full GDPR programme.

**Team support an architect would provide.**
- A weekly architecture-decision review during scale-up; the V / D pattern that produced V1-V6 + D1-D22 in this PoC scales into a steady-state ADR log.
- A named owner per production-extension item -- one item, one owner, one acceptance criterion. The kickoff-resolution shape from this PoC is the template.
- A rolling production-readiness checklist tied to the §1.2 success-criteria table -- as each criterion graduates from "PoC PASS" to "production PASS", the verdict column flips and the next criterion comes into focus.

### §6.5 Tying back to the hypothesis

"Simplicity of such a system" is not only simplicity for the end user. It is also simplicity for the team that maintains the system. The SBB shape -- a small, named set of portable artefacts (CI workflows, JMeter harness, Postman collection, Dockerfile, front-end tooling) plus a documented decision trail -- is the architect's deliverable to future teams. A team that inherits this PoC inherits an SBB they can grow by following the same kickoff-resolution rhythm that produced it. That rhythm is what scales the system without scaling its complexity, and it is what the project-management dimension is for.
